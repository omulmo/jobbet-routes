# System Patterns

## Architecture
- Two CDK stacks: CertificateStack (us-east-1) for ACM cert, AppStack (eu-north-1) for everything else
- Cross-stack reference: certificate ARN exported from CertificateStack, imported by AppStack
- CloudFront single distribution serves both S3 (static) and Lambda (API) via path-based routing
- TR-8 (new): layered architecture — presentation logic in frontend, data-oriented APIs using domain primitives (locations, stops, trips), reusable by other clients/MCP agents

## Design Patterns
- Memory Bank pattern: markdown files in `.kiro/memory-bank/` provide persistent context
- Development workflow: requirements → solution design → implementation → tests → deploy → integration test
- Two-tier deployment: CDK for infrastructure changes, direct AWS CLI for code iteration
- Parallel deploy: `deploy.sh all` runs lambda and frontend deploys concurrently
- Config-driven hostname: CDK context parameter with sensible default
- LOCATIONS dict: static config in Lambda with named locations, each having stops with walk_minutes (will move to persistence layer)

## Conventions
- **Always use `python3`** (never `python`) when executing Python commands
- **Always use a virtual environment (`venv`)** for Python work
- **Always use the project scripts for deployments** — never suggest raw CLI commands when a script exists:
  - `./setup-infrastructure.sh` for CDK/infrastructure deploys
  - `./deploy.sh lambda|frontend|all` for code deploys
- Use `./node_modules/.bin/cdk` instead of `npx cdk` (stdout issue)
- `CDK_DEFAULT_ACCOUNT` derived from `aws sts get-caller-identity`, never hardcoded
- Emoji in Python code: use named `\uXXXX` constants (e.g. `ICON_METRO`, `ICON_BUS`) — never inline emoji literals
- SL API station names: use `leg.origin.parent.disassembledName` for human-readable names at transfer points
