# System Patterns

## Architecture
- Two CDK stacks: CertificateStack (us-east-1) for ACM cert, AppStack (eu-north-1) for everything else
- Cross-stack reference: certificate ARN exported from CertificateStack, imported by AppStack
- CloudFront single distribution serves both S3 (static) and Lambda (API) via path-based routing

## Project Structure
```
kiro-dev-workshop/
├── cdk/                    # CDK infrastructure (TypeScript)
│   ├── bin/app.ts          # App entry, instantiates stacks with hostname context
│   ├── lib/certificate-stack.ts
│   └── lib/app-stack.ts
├── lambda/                 # Python Lambda handler
├── frontend/               # Static HTML + CSS
├── deploy.sh               # Fast code deploys (lambda/frontend/all)
├── setup-infrastructure.sh # CDK bootstrap + deploy + save resource names
├── deploy.env              # Resource names (gitignored, generated by setup script)
└── .amazonq/rules/memory-bank/  # Persistent context files
```

## Design Patterns
- Memory Bank pattern: markdown files in `.amazonq/rules/memory-bank/` provide persistent context
- Two-tier deployment: CDK for infrastructure changes, direct AWS CLI for code iteration
- Parallel deploy: `deploy.sh all` runs lambda and frontend deploys concurrently
- Config-driven hostname: CDK context parameter with sensible default

## Conventions
- Memory Bank files should be updated as the project evolves
- Use `./node_modules/.bin/cdk` instead of `npx cdk` (stdout issue)
- `CDK_DEFAULT_ACCOUNT` derived from `aws sts get-caller-identity`, never hardcoded
